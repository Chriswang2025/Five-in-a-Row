# 五子棋在线对战游戏 - 设计文档

> 设计日期：2026-07-07  
> 项目类型：双人在线对战H5游戏

---

## 🎯 项目目标

开发一个简单的双人在线五子棋游戏，支持iPhone用户通过浏览器直接对战。

**核心特点：**
- 用户路径极简（打开链接 → 自动匹配 → 开始游戏）
- 支持断线重连（30秒容错）
- 随机匹配机制
- 完全免费部署

---

## 👥 用户路径

### **理想流程**
```
用户A打开链接 → 点击"开始游戏" → 等待对手
用户B打开链接 → 点击"开始游戏" → 立刻匹配成功
→ 双方开始下棋 → 显示胜负 → 回到首页
```

### **排队流程**
```
用户A和B正在游戏中
用户C打开链接 → 看到"游戏进行中，请等待..." → 进入等待队列
用户D打开链接 → 看到"当前排队：2人" → 进入等待队列
A和B游戏结束 → C和D自动匹配 → 开始新游戏
```

## 🏗️ 整体架构

### **文字版架构图**
```
用户A的iPhone          云服务器(Render)        用户B的iPhone
    ↓                       ↓                       ↓
Safari浏览器            Node.js后端             Safari浏览器
    ↓                       ↓                       ↓
Vue.js前端  ←WebSocket→  游戏状态管理  ←WebSocket→ Vue.js前端
    ↓                       ↓                       ↓
Canvas棋盘              等待队列                Canvas棋盘
    ↓                       ↓                       ↓
点击下棋  →发送坐标→   验证+转发   →发送坐标→   更新棋盘
```

### **为什么这样设计？**
- **前端（Vue.js）**：负责显示界面和接收用户操作，就像"显示器+鼠标"
- **WebSocket**：保持实时连接，就像两个手机和服务器"一直在打电话"
- **后端（Node.js）**：充当"裁判"，管理游戏状态、验证规则、判断胜负

## 🎮 游戏规则

### **棋盘规则**
- 标准15×15棋盘
- 黑方先手
- 连成5子获胜
- 无禁手（简化规则）

### **匹配规则**
- 随机匹配（先来先配）
- 同时只支持1局游戏
- 其他用户自动进入等待队列

### **断线规则**
- 玩家断线 → 启动30秒倒计时
- 30秒内重连 → 恢复游戏
- 超过30秒 → 对方获胜，游戏结束

## 💻 技术栈

### **前端**
- **框架**：Vue.js 3
- **画布**：HTML5 Canvas（绘制棋盘）
- **通信**：WebSocket（实时对战）
- **部署**：Render.com（免费）

### **后端**
- **运行环境**：Node.js
- **WebSocket库**：ws
- **部署**：Render.com（免费）

### **为什么选这些技术？**
- **Vue.js**：轻量级，上手快，模板语法简单
- **Canvas**：浏览器原生支持，画棋盘性能好
- **WebSocket**：实时双向通信，对手下棋你立刻看到
- **Render**：免费部署，支持WebSocket，自动部署

## 🔧 核心功能模块

### **前端：3个主要页面**

#### **1. 首页（Home.vue）**
**功能：**
- 显示游戏标题和"开始游戏"按钮
- 点击按钮 → 连接WebSocket → 跳转到匹配页

#### **2. 匹配页（Matching.vue）**
**功能：**
- 发送匹配请求：`{type: "join_game"}`
- 显示状态：
  - "等待对手中..." （你是第1个）
  - "游戏进行中，请等待...当前排队：2人" （有人在玩）
- 收到匹配成功 → 跳转到游戏页

#### **3. 游戏页（Game.vue）**
**功能：**
- 显示15×15棋盘（用Canvas画）
- 显示当前状态："轮到你了" / "等待对方"
- 显示断线提示："对手断线中...剩余XX秒"
- 点击空位 → 发送：`{type: "place_stone", data: {x, y}}`
- 收到对方下棋 → 更新棋盘
- 收到游戏结束 → 弹窗显示结果 → 回到首页

---

### **后端：4个核心模块**

#### **1. 游戏状态管理**
**数据结构：**
```javascript
let gameState = {
  status: "idle",              // 状态：idle空闲 / playing游戏中
  playerBlack: null,           // 黑方 {userId, socket, disconnectTimer}
  playerWhite: null,           // 白方 {userId, socket, disconnectTimer}
  board: [],                   // 15×15二维数组（0=空，1=黑，2=白）
  currentTurn: "black",        // 当前轮到谁
  startTime: null              // 游戏开始时间
}
```

**状态流转：**
```
idle（空闲）→ playing（游戏中）→ idle（空闲）
```

#### **2. 等待队列管理**
**数据结构：**
```javascript
let waitingQueue = [];  // [{userId, socket, joinTime}, ...]
```

**队列逻辑：**
- 新用户加入 → 检查游戏状态
  - 游戏空闲 + 队列有1人 → 立刻匹配
  - 游戏空闲 + 队列为空 → 加入队列
  - 游戏进行中 → 加入队列
- 游戏结束 → 从队列头部取2人开始新游戏

#### **3. 连接管理**
**处理逻辑：**
- 用户连接 → 分配唯一ID → 保存到本地存储
- 用户断线 → 启动30秒倒计时
- 用户重连 → 检查是否在游戏中 → 恢复游戏状态
- 超时未重连 → 通知对方获胜 → 游戏结束

#### **4. 游戏逻辑**
**核心功能：**
- 验证落子合法性（是否轮到你、位置是否为空）
- 更新棋盘状态
- 判断胜负（检查横、竖、左斜、右斜4个方向）
- 广播更新给双方

## 📨 前后端通信协议

### **消息格式**
```javascript
{
  type: "消息类型",    // 标识消息用途
  data: { ... }        // 具体数据
}
```

### **协议列表**

#### **连接和匹配阶段**

| 方向 | 消息类型 | 数据 | 说明 |
|------|---------|------|------|
| 前→后 | `join_game` | 无 | 请求加入游戏 |
| 后→前 | `waiting` | `{queuePosition: 1}` | 进入等待队列 |
| 后→前 | `game_busy` | `{message, queuePosition}` | 游戏进行中，需排队 |
| 后→前 | `game_start` | `{color, currentTurn}` | 匹配成功，游戏开始 |

#### **游戏进行阶段**

| 方向 | 消息类型 | 数据 | 说明 |
|------|---------|------|------|
| 前→后 | `place_stone` | `{x, y}` | 下棋 |
| 后→前 | `board_update` | `{x, y, color, currentTurn}` | 更新棋盘 |
| 后→前 | `error` | `{message}` | 非法操作提示 |
| 后→前 | `game_over` | `{winner, reason}` | 游戏结束 |

#### **断线重连阶段**

| 方向 | 消息类型 | 数据 | 说明 |
|------|---------|------|------|
| 后→前 | `opponent_disconnect` | `{remainingTime}` | 对手断线 |
| 前→后 | `reconnect` | `{userId}` | 请求重连 |
| 后→前 | `game_resume` | `{board, color, currentTurn}` | 恢复游戏 |

## 🎨 关键技术实现

### **1. 如何画棋盘？**

**用Canvas绘制15×15棋盘：**
```javascript
// 画网格线
for (let i = 0; i < 15; i++) {
  // 画横线
  ctx.moveTo(起点X, 起点Y + i * 格子大小);
  ctx.lineTo(终点X, 终点Y + i * 格子大小);
  
  // 画竖线
  ctx.moveTo(起点X + i * 格子大小, 起点Y);
  ctx.lineTo(终点X + i * 格子大小, 终点Y);
}

// 画棋子
ctx.arc(圆心X, 圆心Y, 半径, 0, 2π);
ctx.fillStyle = color === 'black' ? '黑色' : '白色';
ctx.fill();
```

**通俗理解：**
- Canvas像一张白纸
- 用循环画线 → 形成网格
- 点击时计算点击位置对应哪个格子
- 在格子中心画圆 → 形成棋子

---

### **2. 如何判断五子连珠？**

**检查4个方向：**
```javascript
// 横向：← 当前棋子 →
// 竖向：↑ 当前棋子 ↓
// 左斜：↖ 当前棋子 ↘
// 右斜：↗ 当前棋子 ↙

function checkWin(x, y, color) {
  const directions = [
    [[0,1], [0,-1]],      // 横向
    [[1,0], [-1,0]],      // 竖向
    [[1,1], [-1,-1]],     // 左斜
    [[1,-1], [-1,1]]      // 右斜
  ];
  
  for (let dir of directions) {
    let count = 1;  // 当前棋子算1个
    
    // 沿两个方向数（比如横向：左+右）
    for (let [dx, dy] of dir) {
      let nx = x + dx, ny = y + dy;
      while (在棋盘内 && board[nx][ny] === color) {
        count++;
        nx += dx;
        ny += dy;
      }
    }
    
    if (count >= 5) return true;  // 5连以上获胜
  }
  return false;
}
```

**通俗理解：**
- 下完一颗棋后，从这颗棋开始
- 横向：往左数+往右数，加起来≥5 → 赢
- 竖向、斜向同理
- 就像"数珠子"，看能不能数到5颗连着的

### **3. 如何处理断线重连？**

**前端：心跳检测**
```javascript
// 每5秒发一次心跳
setInterval(() => {
  if (socket连接正常) {
    socket.send({type: 'ping'});
  } else {
    reconnect();  // 尝试重连
  }
}, 5000);

// 重连时发送之前保存的userId
socket.send({
  type: 'reconnect',
  userId: localStorage.getItem('userId')
});
```

**后端：30秒倒计时**
```javascript
// 用户断线时
socket.on('close', () => {
  // 启动30秒倒计时
  const timer = setTimeout(() => {
    notifyOpponent('对方超时，你赢了');
    gameState.status = 'idle';
  }, 30000);
  
  player.disconnectTimer = timer;
});

// 用户重连时
if (reconnectUser) {
  clearTimeout(player.disconnectTimer);  // 取消倒计时
  sendGameState(socket);                 // 发送当前棋盘状态
}
```

**通俗理解：**
- 前端每5秒"喊一声"，确认连接正常
- 断线立刻尝试重连
- 后端收到断线 → 开始倒计时30秒
- 重连成功 → 取消倒计时，游戏继续
- 超时未连 → 对方获胜

## 🚨 风险与限制

### **1. 服务器单点故障**
**风险：**
- 服务器崩溃 → 所有游戏中断

**应对：**
- 定期备份代码
- 使用Render的健康检查功能
- 监控服务器运行状态

---

### **2. Render免费版休眠**
**限制：**
- 15分钟无活动 → 服务器休眠
- 下次访问需要10秒唤醒

**影响：**
- 游戏进行中不会休眠（WebSocket保持连接）
- 只有长时间无人玩，第一个用户需要等待唤醒

**应对：**
- 如果用户增多，升级到付费版（每月7美元，不休眠）

---

### **3. 并发用户限制**
**限制：**
- 当前设计：同时只支持1局游戏（2个用户）
- 其他用户需要排队

**如果想支持多局同时进行：**
- 需要修改后端架构：
  ```
  gameState = {
    room_1: {玩家A, 玩家B, 棋盘...},
    room_2: {玩家C, 玩家D, 棋盘...},
    ...
  }
  ```
- 从"单房间"改为"多房间管理"

---

### **4. 没有数据持久化**
**限制：**
- 游戏数据存在内存中
- 服务器重启 → 所有数据丢失
- 无法查看历史战绩

**应对：**
- 如果以后想加"战绩排行榜"，需要添加数据库（MySQL/MongoDB）
- 当前版本不影响游戏体验

## 📱 部署方案

### **免费部署到Render.com**

1. 创建GitHub仓库，推送前端代码

## 🎓 学习要点（给产品经理的技术笔记）

### **1. WebSocket vs HTTP**
- **HTTP像发短信**：你问一句，服务器答一句，结束
- **WebSocket像打电话**：一直保持连接，双方随时说话
- 五子棋需要"对方下棋你立刻看到" → 必须用WebSocket

### **2. 前后端分离**
- **前端**：负责"显示"和"用户交互"（Vue.js）
- **后端**：负责"业务逻辑"和"数据管理"（Node.js）
- 就像餐厅：前端是"服务员"，后端是"厨师"

### **3. 状态管理**
- **游戏状态（gameState）**：记录"现在游戏是什么情况"
  - 谁在玩？棋盘长什么样？轮到谁了？
- 就像"游戏存档"，服务器用这个来管理游戏

### **4. Canvas绘图**
- 浏览器提供的画图工具，像一张画布
- 用代码画线、圆、矩形等图形
- 五子棋的棋盘和棋子都用Canvas画
